<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| upstream-to-handler | Untrusted upstream responses are converted into public API output and internal success evidence. |
| handler-to-request-log | Handler-derived evidence is written to in-memory logs visible to admin APIs. |
| handler-to-pool | Handler-derived evidence changes later account routing priority. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-04 | Spoofing | `updateRequestLogContentSuccessIfPresent` | mitigate | Require real output tokens, non-empty trimmed content, reasoning, or structured tool output; HTTP status alone is not accepted. |
| T-04-05 | Tampering | stable fallback paths | mitigate | Keep `markRequestLogStableFallback` forcing content failure and never calling pool success recording. |
| T-04-06 | Information Disclosure | request logs and handler tests | mitigate | Assert only account IDs, mapped models, timestamps, and safe fallback flags; do not read or expose runtime secret files. |
| T-04-SC | Tampering | package installs | accept | No npm/pip/cargo package installs are planned in this phase plan. |
</threat_model>
