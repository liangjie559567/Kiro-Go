# 06-01 Plan: Structured Upstream Error Parser and Reason Mapping

## Goal

Kiro-Go parses upstream failure evidence into a normalized safe object and maps it to stable failure reason codes before falling back to legacy string matching.

## Tasks

1. Add a normalized upstream error type with HTTP status, source, JSON `name/code/reason/message/type`, AWS-style code/type, request ID, retry-after seconds/reset time, redacted summary, and retryability fields.
2. Add parser helpers for HTTP responses, JSON bodies, AWS-style payloads, `Retry-After`, `retry-after-ms`, and safe redaction.
3. Make `pool.ClassifyFailureReason` prefer structured evidence through a small interface while preserving existing string fallback behavior.
4. Update `rateLimitError` or upstream HTTP error construction so 429 and non-429 HTTP failures carry normalized evidence.
5. Add parser and classifier table tests.

## Acceptance

- Structured parser tests cover 429, 403, 500, nested JSON errors, AWS-style error payloads, request ID headers, `Retry-After`, and `retry-after-ms`.
- Secret-like content is redacted from summaries and never includes authorization, cookies, API keys, bearer tokens, or raw secret-bearing headers.
- Required reason scenarios map to exactly one stable reason code.
