# 06-03 Plan: Safe Kiro CLI Diagnostics API, UI, and Audit Evidence

## Goal

Admins can inspect safe Kiro CLI diagnostics without reading secrets or performing unsafe account actions by default.

## Tasks

1. Add a CLI diagnostics service with configurable/default CLI path detection, executable availability, version probing, command-router state summary, and `KIRO_CLI_HOME` presence.
2. Add a fakeable command runner with timeout, bounded output, redaction, and a read-only allowlist: `version`, `whoami`, `doctor`, `diagnostic`, `chat --list-models`, and `settings list`.
3. Report Opus 4.7 model-list state as `present`, `unavailable`, or `unknown` using command evidence.
4. Add admin GET/POST routes for summary and explicit diagnostic command runs, with audit metadata recorded in request-log-style evidence or safe structured logs.
5. Render a lightweight safe diagnostics card in the existing admin UI.
6. Add API tests and UI/static tests for safe rendering and redaction.

## Acceptance

- API tests cover CLI missing, CLI present, timeout, non-zero exit, successful JSON/text output, allowlist rejection, redaction, and Opus 4.7 model-list states.
- No diagnostic path reads token stores, auth databases, browser sessions, keychains, API keys, `data/config.json`, or raw runtime secret files.
- UI renders only safe fields and latest redacted result.
