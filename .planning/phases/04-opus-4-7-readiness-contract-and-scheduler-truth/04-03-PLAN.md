<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| admin-client-to-readiness-api | Authenticated admin clients query readiness state derived from runtime account and request-log data. |
| pool-to-admin-api | Runtime account eligibility and success evidence are serialized into JSON. |
| request-log-to-admin-api | Request-log content/fallback evidence is aggregated into readiness status. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-07 | Information Disclosure | `apiGetFleetReadiness` account rows | mitigate | Emit masked email and safe operational fields only; never serialize tokens, raw config, or secret-bearing files. |
| T-04-08 | Tampering | `safeConcurrency` computation | mitigate | Test exact formula and force `0` for blocked status. |
| T-04-09 | Repudiation | reason codes | mitigate | Emit machine-readable `reasonCodes` from explicit account/admission/breaker conditions for auditability. |
| T-04-10 | Spoofing | content evidence aggregates | mitigate | Separate `contentSuccess` from `stableFallbacks` and empty/failure counters; fallback never contributes to healthy model success. |
| T-04-SC | Tampering | package installs | accept | No npm/pip/cargo package installs are planned in this phase plan. |
</threat_model>
