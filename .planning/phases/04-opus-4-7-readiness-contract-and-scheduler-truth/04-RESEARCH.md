## Risks/Pitfalls

- Duplicating eligibility logic between scheduler preview and fleet readiness risks RDY-04 drift; use one helper or one typed row builder for both surfaces. [CITED: .planning/phases/04-opus-4-7-readiness-contract-and-scheduler-truth/04-CONTEXT.md] [VERIFIED: proxy/ecosystem_ops.go]
- Treating stable fallback HTTP 200 as content success would violate RDY-02 and make sub2api trust a fake healthy signal. [CITED: .planning/REQUIREMENTS.md] [VERIFIED: proxy/request_log.go]
- Recording account-level success from transport success alone would be wrong because successful HTTP status can still contain empty completion or stable fallback content. [CITED: .planning/phases/04-opus-4-7-readiness-contract-and-scheduler-truth/04-SPEC.md] [VERIFIED: proxy/request_log.go]
- Persisting account-level model-success evidence in `data/config.json` would expand the secret-bearing runtime file and may create unnecessary schema churn; prefer in-memory pool state unless restart persistence is explicitly required. [VERIFIED: config/config.go] [ASSUMED]
- Applying success-evidence priority before eligibility checks could route to disabled, cooling-down, breaker-open, token-expired, usage-limited, or model-invisible accounts. [CITED: .planning/phases/04-opus-4-7-readiness-contract-and-scheduler-truth/04-SPEC.md] [VERIFIED: pool/account.go]
- Changing `isBetterCandidateLocked` can alter existing load-balancing semantics, especially round-robin behavior, so tests should pin strategy-specific expectations. [VERIFIED: pool/account.go] [VERIFIED: pool/account_test.go]
- `pool.ModelBlockState` currently aggregates breaker/cooldown state separately from `schedulerPreviewRows`; without row-level breaker exposure, scheduler preview can look eligible while fleet readiness says blocked. [VERIFIED: pool/account.go] [VERIFIED: proxy/ecosystem_ops.go]
- Retry loops already contain stable downstream wait paths and budget reset behavior, so broad rewrites could regress current compatibility; patch small, observable branches instead. [VERIFIED: proxy/handler.go] [VERIFIED: proxy/handler_test.go]
- Streaming tests must distinguish pre-first-token retry from post-SSE-start failure because replay after downstream content risks duplicated client output and corrupted stream state. [CITED: .planning/REQUIREMENTS.md] [VERIFIED: proxy/handler.go]
- Admin UI rows must continue masking emails and must not render tokens, raw headers, CLI paths with sensitive home details, or `data/config.json` content. [CITED: .planning/phases/04-opus-4-7-readiness-contract-and-scheduler-truth/04-UI-SPEC.md] [CITED: .planning/REQUIREMENTS.md]
- Do not copy implementation code, tests, constants, or structure from AGPL `jwadow/kiro-gateway`; only independently derived design ideas are allowed. [CITED: .planning/REQUIREMENTS.md]
