# 05-02 Plan: Gate sub2api Scheduling Before Concurrency Consumption

## Goal

sub2api uses Kiro-Go readiness before consuming account concurrency across sticky, load-balance, and fallback-wait scheduling paths.

## Tasks

1. Identify Kiro-Go candidates using explicit account metadata and OpenAI-compatible base URL matching.
2. Evaluate readiness with the effective upstream model before returning previous-response sticky selections.
3. Evaluate readiness before returning session sticky selections and sticky wait plans.
4. Evaluate readiness before load-balance ordering/acquisition.
5. Evaluate readiness before fallback-wait plans.
6. Treat `blocked` as an exclusion/temp-unschedulable scheduling outcome, not a permanent account error.
7. Treat `degraded` as eligible only within `safeConcurrency` and lower ranked than healthy candidates.

## Acceptance

- Blocked Kiro-Go candidates are skipped without permanent errors.
- Degraded candidates are capped/ranked using `safeConcurrency`.
- Readiness is evaluated before slot acquisition in tested paths.
- Effective upstream model is used, not just the requested model.
